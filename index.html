    Messaging Service

    Send Messages

    To Triggger Message, use this code:

    require 'uri'
require 'net/http'

url = URI("https://api.triggger.co/api/v1/trigggerMessage")

http = Net::HTTP.new(url.host, url.port)

request = Net::HTTP::Post.new(url)
request["content-type"] = 'application/json'
request.body = "{\n\t\"sender\": \"GIT\",\n\t\"target\": \"2348107176019\",\n\t\"payload\": \"Hello boy\",\n\t\"webhook\": \"\",\n\t\"live_key\": \"{apikey}\"\n}"

response = http.request(request)
puts response.read_body

    import requests

    Request Resources

    Parameter Default Description
    Sender true This is the name that appears on the target phone when a message is sent.
    Target true The recipient of the message.
    Payload true This is the body of the message to be sent.
    Webhook false This the webhook URL for recieving realtime message status Learn more.
    Live_key true This is your live apikey. It can be found in your profile in your profile

    Respose Resources

    Parameter Default Description
    Sender true This is the name that appears on the target phone when a message is sent.
    Target true The recipient of the message.
    Body true This is the body of the message to be sent.
    Status true This is the status of the message sent, status ranges from undelivered, sending and delivered.
    Cost true This is the amount of the message sent, usually 0 because we charge for only delivered messages.
    Delivery_time true This is the time the message was sent.
    Message_itinery true This is the time it takes to deliver the message in milliseconds.
    User_id true This is the id of the user initaiting the message
    Country true This is the country where the target service provider operates.
    Message_sid true This is the system identifier for managing messages.
    Hook true This is the bwebhook URL for recieving realtime message status Learn more.
    Hook true This is the length of the message, we consider 150 characters as 1 message.

    Webhooks are user-defined HTTP callbacks. They are usually triggered by some event, such as receiving an SMS message or an incoming phone call. When that event occurs, Twilio makes an HTTP request (usually a POST or a GET) to the URL configured for the webhook.

    To handle a webhook, you only need to build a small web application that can accept the HTTP requests. Almost all server-side programming languages offer some framework for you to do this. Examples across languages include ASP.NET MVC for C#, Servlets and Spark for Java, Express for Node.js, Django and Flask for Python, and Rails and Sinatra for Ruby. PHP has its own web app framework built in, although frameworks like Laravel, Symfony and Yii are also popular.

    Whichever framework and language you choose, webhooks function the same for every Twilio application. They will make an HTTP request to a URI that you provide to Twilio. Your application performs whatever logic you feel necessary - read/write from a database, integrate with another API or perform some computation - then replies to Twilio with a TwiML response with the instructions you want Twilio to perform.
